# Part 1
## Function Templates

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 1.1 - Origine

Sans les templates, il faudrait écrire un `max` pour chaque type :

```cpp
int   max(int a,   int b)   { return a > b ? a : b; }
float max(float a, float b) { return a > b ? a : b; }
// ... repeat forever
```

Les templates permettent au **compilateur de générer** ce code à votre place :

```cpp
template <typename T>
T max(T a, T b) { return a > b ? a : b; }

max(3, 5);         // T deduced as int
max(3.0f, 5.0f);   // T deduced as float
max<double>(1, 2); // explicit instantiation
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 1.2 - Type Deduction Rules

Le compilateur déduit `T` à partir du site d'appel - il n'est pas nécessaire de le spécifier explicitement :

```cpp
template <typename T>
void print(T val);

int x = 5;
print(x);        // T = int
print(3.14f);    // T = float
print("hello");  // T = const char*
```

> L'instanciation explicite (`print<int>(x)`) n'est requise que lorsque la déduction est ambiguë ou que l'on souhaite forcer un type précis.

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 1.3 - Multiple & Non-Type Parameters

**Plusieurs paramètres de type :**
```cpp
template <typename T, typename U>
auto add(T a, U b) -> decltype(a + b) { return a + b; }
```

**Paramètres non-type** - des valeurs figées à la compilation :
```cpp
template <typename T, int N>
std::array<T, N> makeArray() { return {}; }

auto arr = makeArray<int, 10>(); // std::array<int, 10>
```

> `N` est résolu entièrement à la compilation - aucun coût à l'exécution.

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 1.4 - `if constexpr`

Branchement à la compilation à l'intérieur d'un template - **les branches mortes ne sont pas compilées** :

```cpp
template <typename T>
void describe(T val) {
    if constexpr (std::is_integral_v<T>)
        std::cout << "integer: " << val;
    else if constexpr (std::is_floating_point_v<T>)
        std::cout << "float: " << val;
    else
        std::cout << "other";
}
```

> Contrairement à un `if` classique, la branche écartée n'a même pas besoin d'être valide pour le type donné.

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 1.4 - Key Rule: Headers Only

> ⚠️ **L'implémentation des templates doit se trouver dans les fichiers d'en-tête**, pas dans les `.cpp`.

Le compilateur a besoin de la **définition complète** au moment de l'instanciation.

```
// ✅ Correct
template <typename T>        // definition in .h
T max(T a, T b) { ... }

// ❌ Wrong
// max.h  →  declaration only
// max.cpp →  definition        (linker error at use site)
```

## 2.1 - Basic Class Template

```cpp
namespace core {

template <typename T>
class Stack {
public:
    void push(T val)     { data_.push_back(val); }
    T    top()  const    { return data_.back(); }
    void pop()           { data_.pop_back(); }
    bool isEmpty() const { return data_.empty(); }

private:
    std::vector<T> data_;
};

} // namespace core
```

```cpp
Stack<int>         intStack;
Stack<std::string> strStack;
// Two completely separate classes generated by the compiler
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 2.2 - Member Function Templates

Une classe template peut avoir **ses propres** méthodes templates :

```cpp
template <typename T>
class Container {
public:
    template <typename U>
    void copyFrom(const Container<U>& other)
        requires std::convertible_to<U, T>;
};
```

> `T` est le type au niveau de la classe. `U` est propre à cette méthode - il peut différer de `T` tant que `U` est convertible vers `T`.

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 2.3 - Template Specialization

Personnaliser le comportement pour **des types précis** :

https://github.com/SAE-Geneve/CPlusPlus_Course_Template/

```cpp
// Primary template - general case
template <typename T>
class Serializer {
public:
    static std::string serialize(T val) { return std::to_string(val); }
};

// Full specialization for bool
template <>
class Serializer<bool> {
public:
    static std::string serialize(bool val) { return val ? "true" : "false"; }
};
```

## 2.4 - Real Example: `Vec2<T>`

Un vecteur 2D générique - plusieurs fonctionnalités des templates réunies dans une seule classe :

https://github.com/SAE-Geneve/CityBuilder_SFML/blob/master/core/include/maths/vec2.h

```cpp
namespace core {

template <typename T>
    requires std::is_arithmetic_v<T>
class Vec2 {
public:
    T x{}, y{};
    constexpr Vec2() = default;
    constexpr Vec2(T x, T y) : x{x}, y{y} {}

    [[nodiscard]] Vec2 operator+(Vec2 o) const { return {x + o.x, y + o.y}; }
    [[nodiscard]] Vec2 operator-(Vec2 o) const { return {x - o.x, y - o.y}; }
    [[nodiscard]] Vec2 operator*(T s)    const { return {x * s,   y * s};   }

    [[nodiscard]] T sqrLength() const { return x * x + y * y; }

    [[nodiscard]] T length() const
        requires std::is_floating_point_v<T>  // only exists for Vec2<float>
    { return std::sqrt(x * x + y * y); }

    static T dot(Vec2 a, Vec2 b) { return a.x * b.x + a.y * b.y; }
};

using Vec2i = Vec2<int>;
using Vec2f = Vec2<float>;

} // namespace core
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 2.4 - `Vec2<T>` - Key Points

- **Contrainte au niveau de la classe** : `requires std::is_arithmetic_v<T>` - rejette `Vec2<std::string>` dès la compilation
- **Contrainte au niveau de la méthode** : `length()` n'existe que pour `Vec2<float>`, pas pour `Vec2<int>`
- **Alias de type** : `Vec2i`, `Vec2f` - ergonomique, sans coût à l'exécution
- **`[[nodiscard]]`** : le compilateur émet un avertissement si la valeur de retour est ignorée

```cpp
Vec2f a{1.0f, 0.0f};
Vec2f b{0.0f, 1.0f};

auto c = a + b;           // {1, 1}
auto len = c.length();    // ✅ only compiles for Vec2f

Vec2i d{1, 2};
// d.length();            // ❌ compile error - constraint not satisfied
```

## 3.1 - Standard Library Concepts First

Avant d'écrire les vôtres, sachez ce que C++20 propose dans `<concepts>` :

> **Utilisez-les comme briques de base** - composez-les avant d'écrire quoi que ce soit de zéro.

| Concept                         | Signification                              |
| ------------------------------- | ------------------------------------------ |
| `std::same_as<T, U>`            | T et U sont exactement le même type        |
| `std::convertible_to<From, To>` | From est implicitement convertible vers To |
| `std::integral<T>`              | T est un type entier                       |
| `std::floating_point<T>`        | T est un type à virgule flottante          |
| `std::totally_ordered<T>`       | T supporte `<`, `>`, `==`, etc.            |
| `std::default_initializable<T>` | T peut être initialisé par valeur          |
| `std::invocable<F, Args...>`    | F peut être appelé avec Args               |
| `std::ranges::range<T>`         | T possède begin/end                        |

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.2 - Why Concepts?

Avant C++20, contraindre des templates passait par le **SFINAE** - cryptique et illisible :

```cpp
// SFINAE - hard to write, impossible to read
template <typename T, typename = std::enable_if_t<std::is_arithmetic_v<T>>>
T square(T x) { return x * x; }
```

Les concepts apportent **des contraintes lisibles et des messages d'erreur lisibles** :

```cpp
template <typename T>
concept Arithmetic = std::is_arithmetic_v<T>;

template <Arithmetic T>
T square(T x) { return x * x; }

square("hello"); // Error: "hello" does not satisfy Arithmetic
//               // ↑ the compiler tells you exactly why
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.3 - Defining Concepts

**Encapsuler un type trait :**
```cpp
template <typename T>
concept Numeric = std::is_arithmetic_v<T>;
```

**Expression `requires` - contraintes structurelles / d'interface :**
```cpp
template <typename T>
concept Printable = requires(T val) {
    { std::cout << val } -> std::same_as<std::ostream&>;
};
```
---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.4 - Concepts composés
**Concept composé - combiner des concepts existants :**
```cpp
template <typename T>
concept Sortable = requires(T container) {
    { container.begin() } -> std::input_iterator;
    { container.end()   } -> std::input_iterator;
    requires std::totally_ordered<typename T::value_type>;
};
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.5 - Four Ways to Apply Concepts

```cpp
// 1. Requires clause - most explicit
template <typename T>
    requires Numeric<T>
T square(T x) { return x * x; }

// 2. Abbreviated syntax - cleanest ✅ preferred
template <Numeric T>
T square(T x) { return x * x; }

// 3. auto with concept
Numeric auto square(Numeric auto x) { return x * x; }

// 4. Trailing requires
template <typename T>
T square(T x) requires Numeric<T> { return x * x; }
```

> Privilégier le **style 2** pour la cohérence dans toute la base de code.

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.6 - Concepts on `Vec2`: Interop

`Vec2` doit pouvoir accepter des types de vecteurs externes (par exemple ceux d'une bibliothèque de physique).
Deux concepts gèrent cela proprement :

```cpp
// Exact match - external type has x/y of the same type T
template <typename VectorT, typename T>
concept is_vector2 = requires(VectorT v) {
    { v.x } -> std::same_as<T&>;
    { v.y } -> std::same_as<T&>;
};

// Looser match - x/y are convertible to T
template <typename VectorT, typename T>
concept is_convertible_to_vector2 = requires(VectorT v) {
    { v.x } -> std::convertible_to<T>;
    { v.y } -> std::convertible_to<T>;
};
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## 3.7 - Concepts on `Vec2`: Subsumption

Deux constructeurs - un pour la correspondance exacte, un pour le convertible - **sans ambiguïté grâce à la négation** :

```cpp
template <typename VectorT>
    requires is_vector2<VectorT, T>
constexpr explicit Vec2(VectorT v) : x(v.x), y(v.y) {}

template <typename VectorT>
    requires is_convertible_to_vector2<VectorT, T> && !is_vector2<VectorT, T>
constexpr explicit Vec2(VectorT v)
    : x(static_cast<T>(v.x)), y(static_cast<T>(v.y)) {}
```

> `&& !is_vector2<VectorT, T>` évite l'ambiguïté lorsque les deux contraintes pourraient correspondre.
> C'est la **subsomption de concepts** - le compilateur choisit la surcharge la plus contrainte.

# Exercise 1
## Class Template: `AssetManager<TAsset, TAssetType>`
### 30 minutes

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## Exercise 1 - Context

Votre base de code contient `core::experimental::AssetManager`.
Objectif : **le reconstruire à partir de zéro**, comprendre chaque choix de conception, puis l'étendre.

**Cahier des charges :**
- Template sur `TAsset` (par exemple `sf::Texture`) et `TAssetType` (un enum)
- Contraindre `TAssetType` avec un concept `AssetEnum` - doit être un enum doté d'une sentinelle `kLength`
- Stocker les assets dans un `std::array<TAsset, N>` dimensionné par `TAssetType::kLength` - **aucune allocation dynamique**
- Implémenter `load(std::span<const std::string_view>)` pour construire chaque asset à partir d'un chemin
- Implémenter `get(TAssetType)` retournant un `const TAsset&`, et levant `std::out_of_range` si l'indice est invalide
- À placer dans le `namespace core`

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## Exercise 1 - Skeleton

```cpp
#ifndef CORE_ASSET_MANAGER_H
#define CORE_ASSET_MANAGER_H

#include <array>
#include <span>
#include <string_view>
#include <stdexcept>

namespace core {

template <typename TAssetType>
concept AssetEnum = std::is_enum_v<TAssetType> && requires {
    { static_cast<size_t>(TAssetType::kLength) };
};

template <typename TAsset, AssetEnum TAssetType>
class AssetManager {
    static constexpr size_t kSize = static_cast<size_t>(TAssetType::kLength);

public:
    void load(std::span<const std::string_view> paths);

    [[nodiscard]] const TAsset& get(TAssetType index) const;

private:
    std::array<TAsset, kSize> assets_{};
};

} // namespace core
#endif
```

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## Exercise 1 - What to Implement

**`load()`** - peupler `assets_` à partir des chemins :
- Itérer sur `paths`, construire chaque `TAsset` à partir de la chaîne du chemin
- Supposer que `TAsset` est constructible depuis un `std::string_view`

**`get()`** - accès avec vérification des bornes :
- Convertir l'enum en `int64_t`
- Lever `std::out_of_range` si hors bornes
- Retourner un `const TAsset&`

**Vérifier** avec le même montage que votre test existant :
```cpp
enum class TextureIndex { kDefault = 0, kGrass, kMaison, kLength };

core::AssetManager<sf::Texture, TextureIndex> mgr;
static constexpr std::array<std::string_view, 3> paths{{
    "empty.png", "grass.png", "maison.png"
}};
mgr.load(paths);
const sf::Texture& tex = mgr.get(TextureIndex::kMaison);
```

## Exercise 1 - Bonus

Ajouter un paramètre template non-type `StringLiteral` pour un **chemin de dossier de base**, assemblé au chargement :

```cpp
template <size_t N>
struct StringLiteral {
    constexpr StringLiteral(const char (&arg)[N]) { std::copy_n(arg, N, str); }
    char str[N]{};
};

// Usage - folder path baked at compile time
core::AssetManager<sf::Texture, TextureIndex, "../_assets/sprites/"> mgr;
// load() then assembles: "../_assets/sprites/grass.png"
```

> Cela correspond à la version expérimentale déjà présente dans votre base de code.

# Exercise 2

## Exercise 2 - Context

Implémenter un header d'utilitaires mathématiques en respectant les règles du projet.

> `std::lerp` est **interdit** dans votre base de code - implémentez-le vous-même.

**Quatre parties :**
- **A** - `lerp` avec un concept `Lerpable` personnalisé
- **B** - `clamp` à l'aide d'un concept de la bibliothèque standard
- **C** - `map` pour remapper une valeur entre deux intervalles
- **D** - Tester avec `Vec2f`

---
<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## Exercise 2 - Part A & B

**`lerp` - concept personnalisé :**
```cpp
template <typename T>
concept Lerpable = std::is_arithmetic_v<T> || requires(T a, T b, float t) {
    { a + (b - a) * t } -> std::convertible_to<T>;
};

template <Lerpable T>
T lerp(T a, T b, float t);
// formula: a + (b - a) * t
```

**`clamp` - concept standard :**
```cpp
template <std::totally_ordered T>
T clamp(T val, T lo, T hi);
```

> `std::totally_ordered` provient de `<concepts>` - inutile de le définir vous-même.

---

<!-- slide bg="[[01_slide_GP_GA_22_08_22.jpg]]" -->
## Exercise 2 - Part C & D

**`map` - remapping entre intervalles :**
```cpp
template <std::floating_point T>
T map(T val, T inMin, T inMax, T outMin, T outMax);
// formula: outMin + (val - inMin) / (inMax - inMin) * (outMax - outMin)
```

**Test avec `Vec2f` :**

`Vec2<float>` possède déjà `operator+`, `operator-`, `operator*(T)`.
Vérifiez qu'il satisfait `Lerpable`, puis appelez :

```cpp
Vec2f result = lerp(Vec2f{0.0f, 0.0f}, Vec2f{1.0f, 1.0f}, 0.5f);
// expected: {0.5f, 0.5f}
```

> Si `Vec2f` ne compile pas avec `lerp`, vérifiez quel opérateur manque et ajoutez-le.

# Summary

---

## What We Covered

| Thème | À retenir |
|---|---|
| Templates de fonctions | Le compilateur génère le code par type ; préférer la déduction à l'instanciation explicite |
| Templates de classes | Spécialisation pour un comportement personnalisé par type |
| Paramètres non-type | Figer des valeurs (tailles, chaînes) à la compilation |
| Concepts standards | Utiliser `<concepts>` avant d'écrire les vôtres |
| Concepts personnalisés | Contraindre les types clairement ; toujours préférer aux SFINAE |
| `if constexpr` | Branchement à la compilation, remplace les schémas à base de `dynamic_cast` |
| Variadics | Connaître la syntaxe, éviter d'en écrire |
| Règles du projet | Pas de `std::function` / `any` ; utiliser plutôt les concepts + `std::invocable` |
